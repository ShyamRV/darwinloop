# Security Policy

darwinloop is a self-improvement engine that executes LLM-generated code. This document
describes every safety control in place to ensure that no generated code can harm your
system, exfiltrate data, or affect anything outside the designated sandbox.

---

## Safety Architecture

### 1. AST Validation Before Execution

**File**: `src/darwinloop/sandbox/validator.py`

Every code snippet generated by the LLM is parsed with Python's `ast` module **before**
it is ever executed. The following patterns cause immediate rejection (the code is
discarded — never run):

- `eval()` or `exec()` calls
- `compile()` calls
- `subprocess.run(..., shell=True)` or `subprocess.call(..., shell=True)`
- `__import__()` calls

Network imports (`requests`, `socket`, `httpx`, etc.) produce a **warning** but are not
blocking — they will simply fail at runtime inside the sandbox.

### 2. Subprocess Isolation

**File**: `src/darwinloop/sandbox/executor.py`

All agent code runs in a **child process** spawned via `subprocess.run`. The darwinloop
library process itself never executes user-supplied or LLM-generated code directly.
Each run gets:

- A fresh UUID-named temporary directory under the OS temp folder.
- An isolated working directory (never the project root).
- Captured stdout/stderr (capped at 50 000 characters).
- Automatic cleanup of the temp directory on completion.

### 3. Hard Timeouts

Every sandbox execution has a configurable timeout (default: **30 seconds**).
When the timeout expires, the subprocess is killed and the result is marked
`timed_out=True`. The score for that run is 0.0.

Configure via:
```python
DarwinLoop(target="...", sandbox_timeout=60)  # 60s timeout
```

### 4. No Network in Sandbox

Sandbox processes are not explicitly network-sandboxed at the OS level (Docker would be
needed for that), but:

- Network imports are flagged as warnings by the AST validator.
- The sandbox environment has no credentials or tokens injected.
- Generated code that attempts network calls will fail with connection errors.

For production use, we recommend wrapping the sandbox in a Docker container with
`--network=none`.

### 5. Immutable Archive

**File**: `src/darwinloop/core/archive.py`

Once an `AgentEntry` is written to the archive, it is never modified. The `add_agent`
method silently ignores duplicate `agent_id` values. This means:

- Every generation is preserved exactly as it was evaluated.
- You can always roll back by reading any previous generation from the archive.
- The archive JSON file is append-only in practice.

### 6. Diff Transparency

Every accepted improvement is stored as both:

1. A full code snapshot (`AgentEntry.code_snapshot`).
2. A unified diff computed between the parent and child snapshots.

The `EvolutionResult.diff` property always shows you exactly what changed.
Nothing is hidden.

### 7. Score Regression Protection

New code is **only accepted** if its benchmark score strictly exceeds the parent's score.
Regressions are recorded in the archive (for open-ended exploration) but the regressed
code is NOT used as the "current best" and will not be written back by `result.apply()`.

### 8. Dry Run Mode

`MockLLMClient` exercises the full darwinloop pipeline — validation, sandbox execution,
archive, selector — without calling any LLM API and without cost. Use it to verify your
pipeline is working correctly before spending tokens.

```bash
darwinloop evolve agent/router.py --dry-run --auto
```

### 9. Human Checkpoints

In non-`--auto` mode, darwinloop prompts the user to confirm before each iteration:

```
Continue? [y/n]:
```

Press `n` to stop at any point. The archive is saved after every iteration.

### 10. No Secrets in Generated Code

The LLM is never given API keys, database credentials, or other secrets. The
`DarwinLoop` constructor does not inject environment variables into the sandbox.

---

## Reporting Security Issues

If you find a security issue in darwinloop, please open a GitHub issue with the label
`security`. Do not post exploit code publicly.

---

## Threat Model

darwinloop is designed to safely evolve **agent logic files** (routing, classification,
processing code). It is NOT designed to safely evolve:

- Agents that have database write access and execute their own code.
- Agents that run with elevated OS privileges.
- Agents that have access to production credentials at import time.

For those use cases, wrap the sandbox in a Docker container with `--network=none`,
a read-only filesystem, and no mounted credentials.
